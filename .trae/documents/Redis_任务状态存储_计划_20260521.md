# Redis 任务状态存储（Ingest Job Store）计划

## Summary

将抓取入库的任务/状态存储从“单进程内存 dict”升级为 Redis 存储，保持现有路由与前端轮询协议不变；同时抽象出存储层接口，为未来引入独立队列/Worker（RQ/Celery 等）预留扩展点。

额外需求：当用户在登录页提交登录时，如果 Redis 不可用，需要明确提示“Redis 不可用，请联系管理员”，并以弹窗形式展示。

## Current State Analysis

- 任务存储目前为进程内全局变量：
  - [ingest_tasks.py](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/services/ingest_tasks.py) 使用 `_jobs: dict[str, dict[str, Any]] + threading.Lock()`。
  - `create_job/update_job/append_job_log/get_job` 仅在当前进程可见，进程重启即丢失。
- 抓取任务由后台线程执行并实时写入任务状态：
  - [routes/ingest.py](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/routes/ingest.py) 中 `_run_ingest_job` 不断调用 `update_job/append_job_log`。
  - 前端 [ingest.html](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/templates/ingest.html) 每 2 秒轮询 `/ingest/status/<job_id>`，依赖返回字段的类型（数字字段要是 int/float，JS 会参与计算进度百分比）。
- Redis 依赖与配置现状：
  - `pyproject.toml` 已包含 `redis` 依赖（无需额外引入）。
  - Redis 环境变量已在 [config/server.py](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/config/server.py#L21-L27) 组织：`RedisHost/RedisPort/RedisPassword/RedisDB/RedisTTL`（用户确认已配置完成）。
- 登录流程当前仅做 DB 校验：
  - [auth.py](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/routes/auth.py) 的 `POST /login` 不检查 Redis。
  - [login.html](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/templates/login.html) 仅以页面 message 的方式展示错误，没有弹窗能力（项目当前也未引入 Semantic UI JS/jQuery）。

## Goals & Success Criteria

- 任务状态跨进程可见：在多进程部署（如 gunicorn 多 worker）下，`/ingest/status/<job_id>` 能读到正确任务状态，不再出现“任务不存在/状态不更新”。
- 任务状态可跨重启：服务重启后，24 小时内可继续查询任务状态与日志（TTL 兜底）。
- 日志体积可控：单任务日志最多保留 500 条（保留最近 500 条）。
- Redis 不可用时明确提示：
  - 登录页面提交时若 Redis 不可用，提示“Redis 不可用，请联系管理员”，并弹窗展示（不影响其它错误如用户名/密码错误的提示方式）。
- 保持接口兼容：前端仍用现有轮询协议与字段名；后端路由尽量不改动或最小改动。

## Proposed Changes

### 1) 新增 Redis 客户端与健康检查工具

**新增文件**
- `webapp/services/redis_client.py`（建议新建）

**职责**
- 从环境变量读取 Redis 配置（与 `config/server.py` 一致的变量名）。
- 提供 `get_redis()`：返回 `redis.Redis` 客户端（`decode_responses=True`），避免在业务代码中散落连接参数解析逻辑。
- 提供 `ping_or_raise()` / `is_available()`：
  - `ping_or_raise()` 用于登录时快速失败并给出明确错误。
  - 错误信息要可直接展示给用户（例如统一抛出 `RedisUnavailableError("Redis 不可用，请联系管理员")`）。

**关键实现细节**
- `RedisPort/RedisDB/RedisTTL` 需要转换为 `int`。
- TTL 默认 86400 秒（24 小时），作为兜底；每次写入/更新时刷新 TTL，确保最长保留不超过 24 小时。
- 不记录/不打印 Redis 密码。

### 2) 将 ingest_tasks 的存储从内存替换为 Redis（保留原有函数签名）

**修改文件**
- [ingest_tasks.py](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/services/ingest_tasks.py)

**目标**
- 保持 `create_job/update_job/append_job_log/get_job` 的接口不变，让 `routes/ingest.py` 无需（或极少）改动。

**Redis Key 设计（建议）**
- 任务元信息 Hash：`ptk:ingest:job:{job_id}`
- 日志 List：`ptk:ingest:job:{job_id}:logs`

**写入策略**
- `create_job(initial_data)`：
  - 生成 `job_id`（继续使用 uuid4 hex）。
  - `HSET` 写入初始字段；同时初始化日志 list（若 `initial_data` 自带 `logs`）。
  - 对 hash 与 logs key 都设置 `EXPIRE(TTL)`。
- `update_job(job_id, updates)`：
  - `HSET` 做浅更新；并刷新 TTL。
- `append_job_log(job_id, message)`：
  - `RPUSH` 追加日志；
  - `LTRIM -500 -1` 保留最近 500 条；
  - 刷新 TTL。
- `get_job(job_id)`：
  - 读取 hash 与 logs list；
  - 关键点：将 Redis 返回的字符串字段转回正确类型（int/float/bool/None），保证 `jsonify` 返回给前端后仍是数字类型，避免 JS 计算百分比时出现字符串拼接/NaN。

**字段类型转换清单（基于当前任务结构）**
- int 字段：`rows`, `waiting_time`, `max_retries`, `deleted_rows`, `inserted_rows`, `rows_per_page`, `start_page`, `end_page`, `current_page`, `processed_pages`, `total_pages`, `total_records`
- float 字段：`retry_delay_seconds`, `elapsed_seconds`, `last_page_elapsed_seconds`, `base_elapsed_seconds`
- bool 字段：`clear_existing`, `cancel_requested`
- 允许为 None 的字段需处理空字符串/缺失（例如 `total_pages/total_records/current_page/end_page`）。

### 3) 登录时检测 Redis 可用性，并弹窗提示

**修改文件**
- [auth.py](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/routes/auth.py)
- [login.html](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/templates/login.html)

**行为**
- 在 `POST /login` 入口处（优先在用户名密码校验之前）执行 `ping_or_raise()`：
  - 如果 Redis 不可用，直接返回 `render_template("login.html", error="Redis 不可用，请联系管理员", redis_popup=True)`。
- `login.html`：
  - 当 `redis_popup=True` 时，使用原生 `alert(error)` 弹窗（项目未引入 Semantic UI JS/jQuery，原生 alert 最稳妥）。
  - 用户名/密码错误仍维持当前页面 message 形式，不触发弹窗。

### 4) 为未来“队列/Worker”预留扩展点（不在本次落地）

**做法**
- `ingest_tasks.py` 内部可实现为 `JobStore` 抽象（内存/Redis 两个实现），但对外仍暴露相同函数。
- 本次默认走 Redis 实现；未来引入 RQ/Celery 时，任务执行从 Thread 迁移到 Worker，仅需替换“执行器”，状态存储仍复用 Redis JobStore。

## Assumptions & Decisions

- Redis 中不保留永久数据：统一 TTL 24 小时兜底（`RedisTTL`）。
- 单任务日志最多保留最近 500 条。
- Redis 不可用时不做内存降级（避免多进程下出现“状态不一致”）；对用户返回明确可读的错误提示。
- 弹窗提示仅针对 Redis 不可用场景；其它错误仍用页面 message 展示。

## Verification

1) Redis 可用场景（本地）
- 运行 Web：`uv run webapp/app.py`
- 登录成功后进入抓取入库页，提交任务：
  - `/ingest` 返回 job_id
  - `/ingest/status/<job_id>` 返回的数字字段类型正确（例如 `processed_pages` 为 number，前端进度条能正常计算）。
- 任务日志持续追加且最多保留 500 条。

2) Redis 不可用场景
- 停掉 Redis 或改错 `RedisHost/RedisPort` 后重新尝试登录：
  - 登录页面提示“Redis 不可用，请联系管理员”
  - 浏览器出现弹窗提示

3) 多进程一致性（可选）
- 以多进程方式启动（如你已有 gunicorn/其它方式）：启动两个 worker 后重复抓取任务与轮询，确保任一 worker 都能读取到任务状态（不再出现“任务不存在”）。
