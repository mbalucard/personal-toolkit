## 摘要

本方案为现有 Flask + SQLAlchemy 管理后台增加 4 类能力：

1. 登录失败次数限制：同一账号密码连续输错达到 5 次后锁定 2 小时；锁定期间禁止登录；在“用户设置”界面可手动解锁。
2. 管理员权限：在 users 表新增是否管理员字段；仅管理员账号可看到/访问“用户设置”菜单与相关路由；启动时将 admin 用户设为管理员。
3. 软删除：用户设置里的“删除”不再物理删除记录，而是将用户标记为不可用（可用=1，不可用=0）；不可用用户禁止登录。
4. 用户状态筛选：用户设置页面新增“用户状态（可用/不可用）”筛选，默认可用；列表与操作基于筛选结果展示；在不可用列表中提供“恢复”为可用的操作。

额外规则（来自需求确认）：

- 达到第 5 次密码错误即触发锁定（含第 5 次）。
- 需要提供“恢复”为可用按钮。
- 需要在界面支持设置/取消管理员，但只有管理员可操作。
- username=admin 的账号默认管理员且不允许取消管理员（因此也不可被删除/停用）。
- 其他“管理员账号”不允许被删除/停用；必须先取消管理员权限后才能删除/停用。

## 现状分析（基于仓库代码）

- 用户模型：`webapp/models.py` 中 `User` 仅包含 `id/username/name/password_hash/created_at`。
- 登录逻辑：`webapp/routes/auth.py` 的 `login_post()` 仅做密码校验与 `login_user()`，失败直接返回“用户名或密码错误”。
- 用户设置：`webapp/routes/user_mgmt.py` 的 `delete_user()` 目前使用 `session.delete(user)` 做物理删除；`user_settings()` 当前不带筛选条件，直接 `.all()`。
- 导航菜单：`webapp/templates/base.html` 的“用户管理”下拉菜单对所有已登录用户可见。
- 数据库“迁移”方式：`webapp/app.py` 启动时 `Base.metadata.create_all()`，并通过 `webapp/services/bootstrap.py::ensure_user_columns()` 做补列（当前只补 `name` 列）。

## 方案设计

### 1) 数据库字段与模型调整

目标：让锁定、管理员、可用状态全部落库，便于重启后仍生效，并且可以在用户设置界面维护。

**users 表新增字段**

- `is_admin`：是否管理员（boolean，默认 false）。
- `is_active`：是否可用（smallint，1=可用，0=不可用，默认 1）。
- `failed_login_attempts`：连续登录失败次数（smallint，默认 0）。
- `locked_until`：锁定截止时间（timestamptz，可空）。

**涉及文件**

- 更新 `User` ORM 模型字段：`webapp/models.py`
- 启动时补列：扩展 `webapp/services/bootstrap.py::ensure_user_columns()`，对上述新列执行 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...`。
- 管理员初始化：扩展 `webapp/services/bootstrap.py::ensure_admin_user(settings)`：
  - 若 admin 用户不存在：创建时写入 `is_admin=true, is_active=1`。
  - 若 admin 用户已存在：强制更新为 `is_admin=true, is_active=1`（满足“加字段时先把 admin 设置为管理员”）。

### 2) 登录失败锁定逻辑

目标：同账号连续密码错误第 5 次锁定 2 小时；锁定期间禁止登录；成功登录重置计数与锁定状态；不可用用户禁止登录。

**逻辑细节（auth.login_post）**

- 读取 `username/password`，查询 `User`。
- 若用户不存在：返回“用户名或密码错误”（不记录次数，避免为不存在账号造记录）。
- 若 `is_active==0`：返回“账号已停用”。
- 若 `locked_until` 存在且 `locked_until > now()`：返回“账号已锁定，请 2 小时后再试”（可附带剩余时间，作为 UI 提示）。
- 校验密码：
  - 失败：`failed_login_attempts += 1`
    - 若 `failed_login_attempts >= 5`：设置 `locked_until = now() + 2 hours`，并返回“密码错误次数过多，账号已锁定 2 小时”。
    - 否则返回“用户名或密码错误”。
  - 成功：重置 `failed_login_attempts=0`、`locked_until=NULL`，写登录日志，`login_user(user)` 并跳转。

**涉及文件**

- `webapp/routes/auth.py`：修改 `login_post()`。

### 3) 仅管理员可见/可访问“用户设置”

目标：非管理员账号不能看到菜单入口，也不能直接访问 `/users/*` 管理页面（防直链）。

**界面（导航）**

- `webapp/templates/base.html`：
  - “用户管理”下拉菜单仅在 `current_user.is_admin` 为真时渲染。

**后端路由保护**

- `webapp/routes/user_mgmt.py`：
  - 为所有用户管理相关路由（至少 `/users/settings`、`/users/create`、`/users/update/<id>`、`/users/delete/<id>`、`/users/login-logs`）增加管理员校验：
    - `if not current_user.is_admin: abort(403)` 或重定向到业务首页并带 error 提示（二选一，建议 403 更清晰）。
  - 将管理员校验封装为本文件内的小工具函数/装饰器（复用，减少重复）。

### 4) 软删除（停用/恢复）与用户状态筛选

目标：删除=停用（is_active=0）；不可用用户不能登录；默认仅看可用用户；可切换筛选看到不可用；不可用列表允许“恢复”为可用。

**后端**

- `webapp/routes/user_mgmt.py::user_settings()`：
  - 新增 query 参数 `status`（"1"/"0"），默认 "1"。
  - users 查询改为 `session.query(User).filter(User.is_active == status).order_by(User.id.asc())...`
  - 将当前筛选值回传模板用于下拉选中。
- `delete_user()`：
  - 改为软删除：`user.is_active = 0`。
  - 保护逻辑：
    - 不允许停用当前登录用户（保留现有逻辑）。
    - 不允许停用 `username == settings.admin_username`（admin 账号）。
    - 不允许停用 `user.is_admin == True` 的账号（必须先取消管理员后才能停用/删除）。
- 新增 `restore_user()`（建议新增 POST `/users/restore/<id>`）：
  - 将 `is_active` 置回 1。
  - 同样不允许对 admin 做“取消管理员/停用”，但恢复通常不会影响 admin（可保持对称校验）。

**前端（用户设置页）**

- `webapp/templates/user_settings.html`：
  - 增加筛选表单（method=get）：
    - 字段：用户状态（可用=1 / 不可用=0），默认可用
    - 提交后刷新列表（保留 success/error 展示逻辑）
  - 列表增加展示列（至少）：
    - 用户状态（可用/不可用）
    - 管理员（是/否）
    - 锁定状态（未锁定/锁定至某时间）
  - 操作列调整：
    - 可用列表：提供“编辑 / 删除(停用) / 解锁(当锁定时)”（解锁按钮仅在锁定中显示）。
    - 不可用列表：提供“恢复”为可用（并可保留编辑/解锁视需求；建议仅恢复 + 编辑即可）。

### 5) 管理员字段在界面的维护

目标：只有管理员可操作；admin 账号不允许取消管理员；其他管理员账号若需停用，必须先取消管理员后再停用。

**后端**

- `create_user()` / `update_user()` 支持 `is_admin` 字段写入（从表单读取 checkbox）。
- 对 `update_user()` 增加限制：
  - 若目标用户为 admin（`username == settings.admin_username` 或 id 对应 admin）：忽略/拒绝 `is_admin=false` 的提交，并给出错误提示“admin 账号不允许取消管理员权限”。

**前端**

- `user_settings.html`：
  - 创建用户弹窗增加“管理员”勾选框（默认不勾选）。
  - 编辑用户表单增加“管理员”勾选框：
    - 若编辑的是 admin 账号：该勾选框置灰不可编辑，并保持勾选状态。

### 6) 手动解锁

目标：锁定账号可在用户设置中一键解锁，解锁后可立即登录。

**后端**

- 新增 POST `/users/unlock/<id>`：
  - 将 `failed_login_attempts=0`、`locked_until=NULL`。
  - admin 账号允许解锁（不影响管理员属性）。

**前端**

- `user_settings.html` 列表中：
  - 当用户处于锁定中（locked_until > now）显示“解锁”按钮，提交到 `/users/unlock/<id>`。

  - `ensure_user_columns()`：补充 `ALTER TABLE users ADD COLUMN IF NOT EXISTS ...`（4 个新列）。
  - `ensure_admin_user(settings)`：创建/更新时写入 `is_admin=true`，并在既有 admin 用户存在时执行一次补写。

- `webapp/routes/user_mgmt.py`
  - 增加管理员校验（装饰器或工具函数）。

  - “用户管理”菜单仅对管理员展示。

## 假设与决策

- 锁定触发点：连续密码错误次数达到 5 次（含第 5 次）即锁定 2 小时（已确认）。
- 锁定窗口：以 `locked_until` 作为唯一判断；时间到期后不需要后台任务清理，用户再次尝试登录会自然通过锁定校验。
- 软删除语义：is_active=0 的用户禁止登录；仅在“不可用”筛选下可见，并支持“恢复”(已确认)。
- 管理员删除规则：
  - `username=admin` 永远保持管理员且不可被删除/停用。
  - 其他管理员账号默认不可删除/停用，必须先取消管理员后才能执行停用（已确认）。

## 验证步骤（实现后执行）

1. 启动应用并确认启动时自动补列成功（观察数据库 users 表结构或通过页面功能验证）。
   - `uv run webapp/app.py`
2. 管理员可见性：
   - 使用 admin 登录后能看到“用户管理/用户设置”菜单，并可访问 `/users/settings`。
   - 非管理员用户登录后看不到菜单；直接访问 `/users/settings` 返回 403（或被重定向并提示无权限，取决于实现选择）。
3. 登录锁定：
   - 对同一账号连续输错密码 5 次，第 5 次后提示锁定；此后正确密码也无法登录。
   - 在用户设置中点击“解锁”，解锁后可立即使用正确密码登录。
4. 软删除与筛选：
   - 在“可用”列表点击“删除(停用)”后，该用户在可用列表消失，并出现在“不可用”筛选下。
   - 不可用用户无法登录。
   - 在不可用列表点击“恢复”后，用户回到可用列表并可登录。
5. 管理员维护规则：
   - admin 账号在编辑界面无法取消管理员勾选；也无法被删除/停用。
   - 对其他用户：勾选为管理员后，删除/停用按钮应提示需要先取消管理员；取消管理员后才可停用。
