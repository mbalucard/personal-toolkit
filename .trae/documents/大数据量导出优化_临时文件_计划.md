# 大数据量导出优化（临时文件 + 流式写入）计划

## Summary

将当前“内存构建 xlsx → bytes 写入 Response”的导出实现，升级为“数据库结果流式读取 → openpyxl write-only 流式写入 → 落盘到 `webapp/temporary_data` → 浏览器下载”，并按你的要求将临时数据统一放在 `webapp/temporary_data` 下，文件保留 24 小时后自动清理。

## Current State Analysis

当前导出链路（standard/new 两种样式）：

- 路由层：[`webapp/routes/medical_info.py` 的 `export()`](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/routes/medical_info.py#L93-L119)
  - 通过 `export_xlsx(style, version)` 生成 Excel；
  - 以 `Response(file.content)` 返回下载。
- 导出服务层：[`webapp/services/export.py` 的 `export_xlsx()`](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/services/export.py#L108-L138)
  - 先读取 SQL，再调用仓库执行 SQL，得到 `rows`；
  - 使用 `Workbook()`（默认非 write_only）并逐行 `ws.append`；
  - `BytesIO` 保存后 `getvalue()` 返回完整 bytes。
- 仓库层：[`webapp/repositories/drug_update_repo.py` 的 `execute_export_sql()`](file:///Users/lucifer/code/Python/TRAE/personal-toolkit/webapp/repositories/drug_update_repo.py#L116-L129)
  - `fetchall()` 一次性加载所有结果到内存。

对 27 万行（约 20~30 列）来说，风险点：

- `fetchall()` 会把全部结果集载入内存；
- openpyxl 默认模式会创建海量 cell 对象，内存飙升；
- `BytesIO().getvalue()` 让文件二进制再在内存里完整复制一份；
- 导出请求为同步请求，占用 worker 时间较长且并发能力下降。

## Proposed Changes

### 1) 临时目录策略：统一落在 `webapp/temporary_data`

- 目标：所有导出文件落盘到项目目录 `webapp/temporary_data/`。
- 实现要点：
  - 导出前 `mkdir(parents=True, exist_ok=True)`，避免目录不存在导致失败；
  - 文件命名采用“加时间戳”的策略，避免并发冲突：
    - 例如：`20260520_153000_<version>_<style>.xlsx`
  - 增加“保留 24 小时自动清理”：
    - 超时判断不依赖数据库日志表，而是基于文件系统元数据：
      - 读取文件 `mtime`（`Path(file).stat().st_mtime`，即最后修改时间）；
      - 若 `now - mtime > 24h`，则判定为过期并删除；
    - 在每次导出开始时扫描该目录，删除过期文件（等价于“导出触发式清理”）；
    - 仅清理 `.xlsx`（或匹配导出命名模式）以降低误删风险。

涉及文件（执行阶段会修改）：

- `webapp/services/export.py`
  - 新增：临时目录解析/创建、过期清理函数（24h）
  - 新增：生成临时文件路径（时间戳命名）

### 2) 数据库结果改为“流式读取 / 分批读取”

- 目标：避免 `fetchall()` 全量进内存。
- 实现要点（SQLAlchemy 2.0 + PostgreSQL）：
  - 将导出查询执行为“stream_results”模式；
  - 使用 `fetchmany(batch_size=...)` 循环拉取，例如 batch_size=1000/2000；
  - 导出服务边拉边写入 Excel。

涉及文件：

- `webapp/repositories/drug_update_repo.py`
  - 保留现有 `execute_export_sql()`（不影响其他潜在调用）
  - 新增一个导出专用的迭代器函数（例如 `iter_export_sql(...)`），返回一个可迭代的行流（分批 yield）

### 3) Excel 生成改为 openpyxl write-only 流式写入

- 目标：避免 openpyxl 在内存中构建大量 cell 对象。
- 实现要点：
  - `Workbook(write_only=True)`
  - `ws = wb.create_sheet()`
  - `ws.append(columns)` 写表头
  - 对数据库行流逐行 `ws.append(list(row))`

涉及文件：

- `webapp/services/export.py`
  - 将 `export_xlsx()` 改造成“写到指定路径”的函数（例如 `export_xlsx_to_path(...)`）
  - 返回值从 `bytes` 改为 `Path`（或返回 dataclass 含 filename + path）

### 4) 下载响应改为 send_file（从磁盘下载）

- 目标：避免将整个 xlsx bytes 常驻内存。
- 实现要点：
  - 路由层使用 `flask.send_file` 直接发送磁盘文件；
  - `download_name` 使用原文件名（带时间戳）；
  - `mimetype` 保持 xlsx。

涉及文件：

- `webapp/routes/medical_info.py`
  - `export()` 里从调用 `export_xlsx(...)` 改为：
    - 先生成临时文件并得到文件路径
    - `return send_file(path, as_attachment=True, download_name=filename, mimetype=...)`

### 4.1) 临时文件复用：弹窗列出已生成文件并支持直接下载

- 目标：避免重复跑数据库导出；对已生成且仍在保留期内（24h）的文件，允许在页面中直接选择并下载。
- 交互：
  - 在“导出标准”按钮左侧增加一个按钮；
  - 点击按钮弹出小弹窗，展示 `webapp/temporary_data` 下现有的 `.xlsx` 文件列表（文件名、大小、修改时间），并提供“下载”按钮。
- 后端接口（同一 blueprint 下）：
  - `GET /export/files`：返回 JSON 文件列表（同时触发一次过期清理，避免列表堆积旧文件）
  - `GET /export/download/<filename>`：直接下载指定临时文件（做白名单与路径校验，防止路径穿越）
- 前端实现：
  - 使用原生 JS（不引入 jQuery / semantic-ui JS），通过显示/隐藏一个覆盖层 div 实现小弹窗；
  - 通过 `fetch('/export/files')` 拉取列表并渲染；
  - 下载使用普通链接跳转触发浏览器下载。

### 5) Git 忽略策略（避免临时数据被提交）

- 目标：防止临时导出文件进入版本库。
- 实现要点：
  - `.gitignore` 增加 `webapp/temporary_data/`（或 `webapp/temporary_data/*.xlsx`）

涉及文件：

- `.gitignore`

## Assumptions & Decisions

- 临时导出文件需要“保留 24 小时后清理”（已确认）。
- 临时文件名采用“时间戳”方案避免冲突（已确认）。
- 过期判断基于文件系统 `mtime`，不新增“导出记录表/日志表”（如你需要审计追踪，可另行扩展为写入数据库日志表）。
- 导出仍然是同步接口（用户点击导出按钮后等待生成并触发下载）；不引入后台任务/轮询（如后续需要再扩展）。
- 不修改 `api/medical_info.py`（项目硬约束）。
